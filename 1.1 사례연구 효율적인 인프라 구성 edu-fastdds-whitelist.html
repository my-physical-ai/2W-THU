기반 인프라 및 환경 — 왜 Pi5는 가볍게, NUC는 무겁게 구성하는가? DDS와 웹 서비스의 분리, MediaMTX WebRTC의 동작 원리, 그리고 미래의 자동화 모니터링까지 알아봅니다.

🗺️ 전체 아키텍처 — Pi5는 가볍게, NUC는 무겁게
🏗️
건설 현장 비유

Pi5 = 건설 현장의 크레인 운전석 — 현장에서 직접 철근을 들어올리고 내려놓는 실시간 작업만 합니다. 여기에 사무실 업무까지 시키면 크레인이 멈춥니다.

NUC = 현장 사무소 — CCTV 영상 변환, 작업 일지 기록, 외부 보고서 작성 등 "무거운 관리 업무"를 전부 맡습니다. CPU 여유가 충분하니까요.

🍓 왜 Pi5는 가볍게 유지해야 하나요?
💀
실제로 겪은 문제

초기에 Pi5에서 GStreamer RTSP + MediaMTX + ffmpeg를 동시에 실행했습니다.
결과: CPU 100% → Pi5 다운 → 로봇 정지.

Pi5는 ARM 4코어 8GB. 카메라 3대의 H.264 인코딩만 해도 CPU 60~70%를 씁니다. 여기에 프로토콜 변환(WebRTC)까지 올리면 모터 제어 명령이 지연되어 로봇이 반응을 멈춥니다.

Pi5에서 실행하는 것은 딱 2개뿐입니다:

카메라 영상을 GStreamer로 H.264 인코딩하여 RTSP 스트림으로 내보냅니다.

RealSense Depth:8554/depth
USB 카메라:8554/usb

🦿 lerobot_bridge.py

Pi5에 절대 올리면 안 되는 것: MediaMTX(WebRTC 변환), rosbridge(DDS→WS 변환), 대시보드 서버(HTTP). 이것들은 CPU를 많이 잡아먹어 모터 제어 루프(50Hz)를 방해합니다.
📹 MediaMTX WebRTC — iframe으로 카메라 보기
📺
TV 중계차 비유

Pi5 = 현장 카메라맨 — 촬영하고 원본 영상을 케이블(RTSP)로 보냅니다.
NUC MediaMTX = 중계차 — 원본을 받아서 TV 방송 신호(WebRTC)로 변환합니다.
브라우저 iframe = TV 화면 — 시청자는 TV를 켜면 바로 봅니다.

카메라맨(Pi5)에게 "방송 송출까지 해!"라고 하면 과부하. 중계차(NUC)가 변환을 담당해야 합니다.

RTSP → WebRTC 변환이 왜 필요한가?

브라우저는 RTSP 프로토콜을 직접 재생할 수 없습니다. VLC 같은 전용 플레이어만 가능합니다. 대시보드에서 카메라를 보려면 브라우저가 이해하는 프로토콜로 변환이 필요합니다:

프로토콜 브라우저 지원 지연 용도
RTSP (Pi5 원본) ❌ 불가 - VLC, ffplay 전용
WebRTC (:8889) ✅ 네이티브 <1초 대시보드 최적 (권장!)
HLS (:8888) ✅ 네이티브 5~10초 호환성 높지만 지연 큼

💻 대시보드에서의 코드
// MediaMTX WebRTC 페이지를 iframe으로 임베드
// MediaMTX가 자체적으로 WebRTC 플레이어를 제공하므로
// 별도의 WebRTC 라이브러리가 필요 없습니다!

const iframe = document.createElement('iframe');
iframe.src = 'http://10.10.1.20:8889/rs_rgb';  // NUC MediaMTX
iframe.style = 'width:100%; height:100%; border:none';
iframe.allow = 'autoplay';
cameraBox.appendChild(iframe);

// URL 패턴:
//   http://NUC_IP:8889/rs_rgb    ← RealSense RGB
//   http://NUC_IP:8889/rs_depth  ← RealSense Depth
//   http://NUC_IP:8889/cam_usb   ← USB 카메라
💡
왜 iframe인가? MediaMTX가 :8889/스트림명 URL에 이미 완전한 WebRTC 플레이어(HTML+JS)를 제공합니다. 직접 WebRTC API를 구현할 필요 없이, iframe으로 그 페이지를 넣으면 됩니다. WebRTC 연결 협상(SDP, ICE), 코덱 처리를 MediaMTX가 전부 알아서 합니다.
🌉 lerobot_bridge + rosbridge — 왜 이렇게 나누나요?
🏭
공장 비유

lerobot_bridge (Pi5) = 공장 현장의 기계 조작원. 직접 기계(모터)를 돌리고, 센서값을 읽어서 공장 내부 무전(DDS)으로 보고합니다.

rosbridge (NUC) = 공장 사무실의 통역관. 현장 무전(DDS)을 받아서 외부 보고서 형식(WebSocket JSON)으로 번역합니다. 사장님(브라우저)은 보고서만 읽으면 됩니다.

💡 왜 분리? — 조작원(Pi5)에게 "통역도 해!"라고 하면, 기계 조작이 느려집니다. 모터 제어는 50Hz(20ms마다)로 돌아야 하는데, JSON 변환+WebSocket 관리까지 시키면 타이밍을 놓칩니다.

🔄 왜 DDS에서 직접 웹을 지원하지 않나요?
🏛️
DDS의 설계 철학

DDS는 군사·항공·산업용으로 설계된 통신 미들웨어입니다. 전투기 간 통신, 발전소 제어, 수술 로봇 같은 환경에서 마이크로초 단위의 실시간성과 신뢰성이 최우선입니다.

브라우저는 이런 환경의 고려 대상이 아니었습니다. 웹 브라우저는 "HTTP 요청을 보내고 HTML을 받는" 도구이지, "UDP 멀티캐스트를 구독하는" 도구가 아니니까요.

DDS가 웹을 직접 지원하지 않는 기술적 이유:

📡 DDS (로봇 세계)
전송 프로토콜UDP 멀티캐스트
메시지 형식CDR (바이너리, 압축)
DiscoverySPDP/SEDP (자동 탐색)
네트워크같은 서브넷 (LAN)
보안DDS Security 플러그인
지연마이크로초~밀리초
🌐 브라우저 (웹 세계)
전송 프로토콜TCP (HTTP/WebSocket)
메시지 형식JSON (텍스트, 사람 읽기 가능)
Discovery없음 (URL 직접 입력)
네트워크인터넷 어디서든
보안TLS/HTTPS
지연밀리초~초
💡
근본적인 차이: DDS는 UDP 멀티캐스트 + 바이너리 CDR을 쓰고, 브라우저는 TCP WebSocket + JSON 텍스트를 씁니다. 프로토콜, 메시지 형식, 네트워크 모델이 전부 다릅니다. rosbridge는 이 둘 사이의 "동시통역사"입니다.

맞습니다! OMG(DDS 표준 기구)에서 시도한 것들이 있습니다:

기술상태한계
DDS-XRCE (Micro-ROS)Arduino/MCU용브라우저 지원 아님. 마이크로컨트롤러 전용
DDS Web (eProsima)실험적브라우저에서 WASM으로 DDS 구현. 아직 프로덕션 수준 아님
Zenoh (Eclipse)차세대 대안DDS+웹 통합. ROS2 Iron부터 실험 지원. 아직 메인스트림 아님

현 시점에서 가장 안정적이고 검증된 방법은 rosbridge(DDS→WebSocket 변환)입니다. 향후 Zenoh가 성숙하면 rosbridge 없이도 브라우저 연결이 가능해질 수 있습니다.

⚡ NUC의 프로토콜 변환 — 정말 무거운가?

Pi5에서 NUC로 옮긴 작업들이 실제로 얼마나 CPU를 먹는지 비교해봅시다:

📊 작업별 CPU 사용량 비교
작업 Pi5 (ARM 4코어) NUC (x86 멀티코어) 왜 차이?
RTSP→WebRTC (MediaMTX, 3스트림) 30~40% 5~8% WebRTC DTLS 암호화 + SRTP 패킹. x86에 하드웨어 가속
DDS→WebSocket (rosbridge) 10~15% 2~3% CDR→JSON 직렬화. ARM에서는 부동소수점 변환 느림
HTTP 서버 (Phase 1 대시보드) 5~10% 1~2% FastAPI + WebSocket 다중 클라이언트. SQLite I/O
합계 45~65% 8~13%
+ H.264 인코딩 (필수) +60~70% - 인코딩은 Pi5에서만 (카메라 직결)
최종 Pi5 부하 전부 합치면 105~135% → 💀 분리 시 60~70%만
⚠️
핵심: 각 작업이 "무겁다"기보다는, Pi5의 제한된 CPU(ARM 4코어)에 전부 올리면 합산 100%를 넘겨서 문제가 됩니다. NUC는 x86 멀티코어라 같은 작업을 10% 미만으로 처리합니다. 역할 분담의 핵심은 "Pi5에 여유를 주는 것"입니다.

🔮 미래 — 테스트 콘솔에서 자동 모니터링으로

진화 방향

현재 테스트 콘솔은 사람이 직접 조이스틱을 움직이고, 센서값을 눈으로 확인합니다. 하지만 실제 운영 환경에서는 사람이 24시간 화면을 쳐다보고 있을 수 없습니다.

미래에는 서버가 스스로 상태를 체크하고, 이상이 감지되면 관리자에게 알림을 보내는 자동화 파이프라인으로 진화합니다.
