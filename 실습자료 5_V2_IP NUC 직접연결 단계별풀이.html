📋
네트워크 인터페이스 확인
NUC에 어떤 네트워크 포트가 있는지 알아내기
🤔 왜 이 단계가 필요한가요?
NUC에는 여러 개의 네트워크 포트가 있습니다 — 이더넷(랜선) 포트, 내장 WiFi, USB WiFi 등. 이 중에 "카메라 랜선을 꽂을 이더넷 포트"의 정확한 이름을 알아내야 합니다. 이름은 NUC마다 다를 수 있어요. 흔한 이름은 eno1, eth0, enp3s0입니다. 이 이름을 알아야 다음 단계들에서 "이 포트를 설정해줘"라고 시스템에 정확히 지시할 수 있습니다.
⌨️ 터미널에 입력
$nmcli device status
📖 명령어 분해 — 단어별 의미
nmcliNetworkManager Command Line Interface의 줄임말. 우분투의 네트워크 설정을 명령어로 조작할 수 있는 도구입니다.
device"장치"를 다루겠다는 의미. NetworkManager가 인식하는 모든 네트워크 장치 정보를 가져옵니다.
status"상태"를 보여달라는 의미. 각 장치가 현재 어떤 상태인지(연결됨/끊김/사용불가) 출력합니다.
🖥️ 예상 출력 (NUC마다 약간 다름)
DEVICE TYPE STATE CONNECTION wlp0s20f3 wifi 연결됨 NTH215 wlx002e2d40106e wifi 연결됨 zeta_amr_rnd 3 eno1 ethernet 사용할 수 없음 -- lo loopback 연결됨 (외부) lo p2p-dev-wlp0s20f3 wifi-p2p 연결 해제 --
💡 출력을 읽는 법
  • 4개의 열로 출력됩니다: DEVICE(장치 이름) / TYPE(종류) / STATE(상태) / CONNECTION(연결 프로필명)
  • 우리가 찾는 것은 TYPE이 'ethernet'인 줄입니다. 위 예시에서는 eno1입니다.
  • STATE는 지금은 '사용할 수 없음'일 수 있어요 — 케이블이 안 꽂혀 있어서 그렇습니다. 정상입니다.
  • WiFi 두 개(wlp0s20f3, wlx002e2d40106e)는 무시해도 됩니다. 우리는 유선 직결이니까요.
📝
중요 — 메모하세요: 본인 NUC의 이더넷 인터페이스 이름을 적어두세요. 다음 단계들에서 계속 사용합니다. 예: eno1

2
🔍
카메라 IP 자동 발견
tcpdump으로 카메라가 보내는 신호를 엿듣기
🤔 왜 이 단계가 필요한가요?
카메라 IP를 모르면 카메라와 대화할 수 없습니다. 일반 IP 스캐너는 "같은 대역의 IP"만 찾을 수 있어서, 카메라가 우리 NUC과 다른 대역(예: 192.168.1.123 vs 192.168.0.x)이면 발견하지 못합니다. tcpdump은 다릅니다 — 인터페이스를 지나가는 모든 패킷을 가로채서 읽을 수 있어요. 카메라가 부팅하면서 자기 IP를 외치는 신호(ARP/DHCP)를 엿듣고 IP를 알아냅니다.
📻
쉽게 말하면

파티에서 이름 모르는 사람의 이름을 알아내는 방법: 본인이 다른 사람에게 자기소개하는 순간을 옆에서 엿듣기. 카메라는 부팅하면서 "저는 IP 192.168.1.123이에요!"라고 자동으로 외칩니다.

⚠️
사전 준비 (먼저 하세요):
① 카메라 12V 어댑터를 카메라에 연결
② 랜선을 카메라 RJ45 ↔ NUC eno1 양쪽에 꽂기
③ 콘센트에 어댑터 꽂기 (전원 ON)
④ RJ45 LED 녹색이 깜빡이는지 확인 (살아있다는 증거)
⌨️ 터미널에 입력 (eno1은 본인 인터페이스 이름)
$sudo tcpdump -i eno1 -n -e 'arp or (udp port 67 or udp port 68)'
📖 명령어 분해 — 단어별 의미
sudo"관리자 권한으로 실행"하는 명령. tcpdump은 네트워크 카드를 직접 읽어야 해서 권한이 필요합니다.
tcpdump네트워크 패킷을 가로채서 보여주는 도구. "TCP를 떠넘긴다"의 줄임말이지만 모든 패킷을 봅니다.
-i eno1"interface"의 약자. 어느 네트워크 포트에서 들을지 지정. 단계 1에서 확인한 이름 입력.
-n"이름 변환을 하지 마라"는 옵션. IP를 도메인 이름으로 바꾸지 않고 그대로 보여줍니다 (더 빠름).
-e"이더넷 헤더"도 보여달라는 옵션. MAC 주소까지 출력됩니다 (카메라 MAC을 알 수 있음).
'arp or...'필터 표현식. "ARP 패킷이거나 DHCP 포트(67, 68) 패킷만" 보여달라는 의미. 카메라가 부팅 시 보내는 신호입니다.
💡
팁: 명령어 입력 후 잠시 기다려도 출력이 안 나오면 → 카메라 전원을 잠시 끄고 다시 켜세요. 카메라가 부팅하면서 즉시 ARP/DHCP를 보내므로 그 순간을 잡을 수 있습니다.

그리고 약 30초 후 Ctrl+C로 tcpdump을 중단하면 됩니다.
🖥️ 예상 출력 (한 줄이라도 나오면 성공)
listening on eno1, link-type EN10MB (Ethernet), snapshot length 262144 bytes 14:23:01.234567 f0:00:00:99:e0:85 > ff:ff:ff:ff:ff:ff, ARP, Request who-has 192.168.1.1 tell 192.168.1.123, length 46 14:23:01.234890 f0:00:00:99:e0:85 > ff:ff:ff:ff:ff:ff, BOOTP/DHCP, Request from f0:00:00:99:e0:85, length 300 ^C 2 packets captured
💡 출력을 읽는 법 — 한 줄씩 분석
  • 첫 줄(listening on...): tcpdump이 정상 시작했다는 안내. 무시.
  • "f0:00:00:99:e0:85": 분홍색 부분 = 카메라 MAC 주소. 메모해두세요.
  • "tell 192.168.1.123": 노란색 부분 = 카메라 IP 주소. ⭐ 핵심!
  • "ARP, Request who-has X.X.X.X tell Y.Y.Y.Y": Y가 카메라 IP, X는 카메라가 찾는 게이트웨이.
  • "BOOTP/DHCP, Request from MAC": 카메라가 라우터를 찾고 있다는 신호 (직결이라 응답은 없음).
✅ 성공 (출력에 IP가 보임)
메모하세요:
▸ 카메라 IP: __________________
▸ 카메라 MAC: __________________

→ 단계 3으로 진행
❌ 실패 (아무 출력 없음)
점검할 점:
  • RJ45 LED 녹색 깜빡임 확인
  • 인터페이스 이름이 정확한지
  • 카메라 전원 OFF→ON 후 재시도
  • sudo ip link set eno1 up 시도
3
📍
현재 NUC IP 상태 확인
NUC이 지금 어떤 IP를 갖고 있는지 알아내기
🤔 왜 이 단계가 필요한가요?
카메라와 통신하려면 NUC과 카메라가 "같은 동네(같은 IP 대역)"에 있어야 합니다. 카메라가 192.168.1.123에 살고 있다면, NUC도 192.168.1.X(예: 192.168.1.50)에 있어야 통신이 가능해요. 이 단계는 NUC의 현재 IP가 어떤지 확인하여 "이미 같은 대역인지, 아니면 변경이 필요한지"를 판단하기 위함입니다. 대부분의 경우 NUC의 이더넷 포트는 IP가 없거나 다른 대역이므로 다음 단계(4)가 필요합니다.
⌨️ 터미널에 입력 (eno1은 본인 인터페이스)
$ip addr show eno1
📖 명령어 분해 — 단어별 의미
ip리눅스의 강력한 네트워크 관리 명령어. IP 주소, 라우팅, 인터페이스 등 모든 네트워크 작업의 기본.
addr"address"의 줄임. "주소 관련 작업을 하겠다"는 의미.
show"보여달라"는 의미. (다른 옵션: add는 추가, del은 삭제)
eno1어느 인터페이스의 정보를 보여줄지 지정. (생략하면 모든 인터페이스 표시)
🖥️ 예상 출력 (3가지 케이스)

케이스 A: IP가 없음 (가장 흔함)

2: eno1: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 link/ether 1c:69:7a:6f:45:01 brd ff:ff:ff:ff:ff:ff # inet 줄이 없음 → IP가 안 설정된 상태

케이스 B: 다른 대역 IP가 있음

2: eno1: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 link/ether 1c:69:7a:6f:45:01 brd ff:ff:ff:ff:ff:ff inet 192.168.50.100/24 brd 192.168.50.255 scope global eno1 # 카메라는 192.168.1.123인데 NUC은 192.168.50.x → 다른 대역

케이스 C: 이미 같은 대역

2: eno1: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 link/ether 1c:69:7a:6f:45:01 brd ff:ff:ff:ff:ff:ff inet 192.168.1.50/24 brd 192.168.1.255 scope global eno1 # 카메라(192.168.1.123)와 같은 대역 → 단계 4 건너뛰고 7로
💡 출력을 읽는 법
  • "inet" 줄이 있으면 → IP가 설정되어 있음
  • "inet" 줄이 없으면 → IP가 없는 상태 (단계 4에서 설정해야 함)
  • "inet X.X.X.X/24"에서 앞 3자리(X.X.X)가 카메라와 같아야 통신 가능
  • "/24"는 서브넷 마스크 = 같은 동네의 범위 (1~254번 집까지 같은 동네)
🏘️
쉽게 말하면

"192.168.1.123"이 카메라 집 주소라면 "192.168.1.x"는 같은 동네입니다. 카메라와 같은 동네에 있어야 직접 대화할 수 있어요. 다른 동네면 우체국(라우터)을 통해야 하는데, 직결에는 우체국이 없습니다.

✅ 케이스 C: 이미 같은 대역
운이 좋습니다! 단계 4, 5, 6 모두 건너뛰고
→ 단계 7 (ping 확인)로 직행
⚠️ 케이스 A/B: IP 없거나 다른 대역
대부분의 경우 여기 해당.
→ 단계 4로 진행해서 NUC IP 변경 필요
4
⚙️
NetworkManager 영구 고정 IP 설정
⭐ 가장 중요한 단계 — 재부팅에도 유지되는 설정
🤔 왜 이 단계가 가장 중요한가요?
NUC의 IP를 카메라와 같은 대역(예: 192.168.1.50)으로 바꿔야 통신할 수 있어요. 하지만 그냥 IP를 추가하면 NetworkManager가 곧 덮어씁니다(특히 재부팅 시). 그래서 우리는 NetworkManager 자체에 "이 인터페이스는 항상 192.168.1.50을 사용해라"를 영구 등록합니다. 한 번 등록하면 재부팅 후에도 자동으로 적용됩니다.
⚠️
흔히 하는 실수: sudo ip addr add 192.168.1.50/24 dev eno1
이 명령어는 즉시 작동하지만 NetworkManager가 곧 덮어씁니다. 재부팅하면 사라져요. 그래서 우리는 nmcli connection modify를 씁니다.
⌨️ 터미널에 입력 (한 줄로 길게)
$sudo nmcli connection modify "유선 연결 1" \ ipv4.method manual \ ipv4.addresses 192.168.1.50/24 \ ipv4.gateway "" \ ipv4.dns ""
💡
한 줄 vs 여러 줄: 위 명령어는 \(역슬래시)로 줄을 이어붙인 형태입니다. 실제로는 모두 한 명령어예요. 복사하면 자동으로 한 줄로 됩니다. 직접 입력할 때는 \ 다음에 Enter를 누르면 됩니다.
📖 명령어 분해 — 단어별 의미
nmcli connection modify"NetworkManager의 연결 프로필을 수정하겠다"는 의미. 단순히 IP를 임시로 바꾸는 게 아니라 NetworkManager 데이터베이스를 영구 변경.
"유선 연결 1"수정할 프로필의 이름. 단계 1의 출력에서 CONNECTION 컬럼에 있던 이름. 한글 그대로 입력하되 반드시 따옴표로 감쌉니다.
ipv4.method manual"IP를 자동(DHCP)이 아닌 수동으로 설정하겠다"는 의미. auto면 DHCP, manual이면 직접 지정.
ipv4.addresses 192.168.1.50/24NUC이 가질 IP 주소. /24는 서브넷 마스크 = "같은 대역의 범위는 1~254번까지".
ipv4.gateway ""게이트웨이 = "외부로 나갈 때 거치는 라우터". 직결이라 라우터가 없으니 빈 문자열로 둡니다. 인터넷은 WiFi가 담당.
ipv4.dns ""DNS 서버 = "도메인을 IP로 변환해주는 서버". 직결이라 DNS도 빈 값. 인터넷은 WiFi가 담당.
🖥️ 예상 출력 (성공)
# 출력이 거의 없음 — 무소식이 희소식 # (명령이 성공하면 그냥 다음 프롬프트로 돌아옴)
🖥️ 예상 출력 (실패 — 프로필 이름 오류)
Error: unknown connection '유선 연결1'. # 프로필 이름에 공백 누락 → "유선 연결 1"로 정확히
💡 출력을 읽는 법
  • 아무 출력이 없으면 → 성공 (유닉스 철학: 성공은 조용히)

🤔 왜 두 번 명령어를 입력하나요?
단계 4에서는 NetworkManager의 "설정 파일"만 바꿨습니다. 실제 IP는 아직 적용되지 않은 상태예요. 적용하려면 연결을 한 번 끊고(down) 다시 연결(up)해야 합니다. 새 설정을 메모리에 다시 읽어들이기 위함입니다. (재부팅하지 않고도 새 설정을 즉시 적용하는 방법)

▶ 첫 번째 명령어 — 연결 끊기

⌨️ 터미널에 입력
$sudo nmcli connection down "유선 연결 1"
📖 명령어 분해

🖥️ 예상 출력

▶ 두 번째 명령어 — 연결 켜기 (1초 정도 대기 후)

⌨️ 터미널에 입력

📖 명령어 분해

🖥️ 예상 출력 (실패)

💡 두 번째 명령어가 중요

  • "activation failed" → 케이블 또는 프로필 문제. 단계 4 다시 확인
  • 첫 번째 "down" 명령어의 "is not active" 메시지는 무시해도 됨 (이미 꺼져있던 것뿐)
  • 두 명령어 사이에 1초 정도 기다리면 안정적 (시스템에 여유)

IP 적용 결과 확인
설정한 IP가 진짜로 들어갔는지 검증
🤔 왜 또 확인하나요?
단계 5의 "activated" 메시지가 나왔어도 실제 IP가 정확히 적용됐는지는 별개의 확인이 필요합니다. "내가 시킨 대로 됐는가"를 두 번 검증하는 것이 안전합니다. 특히 NetworkManager는 가끔 DHCP를 우선시하거나, 다른 설정이 충돌해서 우리가 원한 IP가 안 들어갈 수 있어요.
⌨️ 터미널에 입력
$ip addr show eno1 | grep 'inet '
📖 명령어 분해 — 새 개념 등장
ip addr show eno1단계 3과 동일 — eno1의 모든 정보 출력.
|"파이프(pipe)" 기호. 앞 명령의 출력을 뒤 명령의 입력으로 넘기는 리눅스의 강력한 기능.
grep 'inet '"inet "이라는 글자가 포함된 줄만 골라서 보여달라는 의미. 'inet '의 공백 한 칸이 중요(IPv6는 'inet6'이라 제외하기 위함).
🖥️ 예상 출력 (성공)
inet 192.168.1.50/24 brd 192.168.1.255 scope global noprefixroute eno1
🖥️ 예상 출력 (실패 — IP 없음)
# 아무 출력이 없으면 → IP 적용 실패 # 또는 다른 IP가 보이면 → 다른 설정이 충돌
💡 출력을 읽는 법 — 한 단어씩 분석
  • "inet 192.168.1.50/24": IPv4 주소 + 서브넷 마스크. 우리가 원한 IP ✅
  • "brd 192.168.1.255": broadcast 주소. /24 대역의 마지막 IP (자동 계산됨)
  • "scope global": 이 IP가 외부 통신에도 사용 가능하다는 의미
  • "noprefixroute": 다른 라우팅을 자동으로 안 만든다는 의미. 직결에 적합
  • 맨 끝 "eno1": 이 IP가 속한 인터페이스 이름
✅ 성공: 192.168.1.50/24 표시

카메라 ping 통신 확인
NUC ↔ 카메라가 실제로 대화 가능한지 검증
🤔 왜 ping이 중요한가요?
ping은 네트워크 통신의 가장 기본적인 테스트입니다. "잘 들리니?"라고 메아리 신호를 보내고 카메라가 "응 들려"라고 답하는지 확인하는 것이죠. ping이 안 되면 그 위에 어떤 것(웹, RTSP)도 안 됩니다. 반대로 ping이 되면 모든 가능성이 열려요.
⌨️ 터미널에 입력
$ping -c 5 192.168.1.123
📖 명령어 분해
ping"Packet Internet Groper"의 약자. 작은 신호 패킷을 보내고 응답을 측정하는 도구. 잠수함 소나에서 유래.
-c 5"count의 약자, 5번만 보내고 종료". 안 쓰면 무한 반복(Ctrl+C 필요).
192.168.1.123대상 IP — 단계 2에서 발견한 카메라 IP.
🖥️ 예상 출력 (정상 — 통신 성공)
PING 192.168.1.123 (192.168.1.123) 56(84) bytes of data. 64 bytes from 192.168.1.123: icmp_seq=1 ttl=64 time=0.5 ms 64 bytes from 192.168.1.123: icmp_seq=2 ttl=64 time=0.4 ms 64 bytes from 192.168.1.123: icmp_seq=3 ttl=64 time=0.5 ms 64 bytes from 192.168.1.123: icmp_seq=4 ttl=64 time=0.4 ms 64 bytes from 192.168.1.123: icmp_seq=5 ttl=64 time=0.5 ms --- 192.168.1.123 ping statistics --- 5 packets transmitted, 5 received, 0% packet loss, time 4081ms rtt min/avg/max/mdev = 0.4/0.5/0.5/0.05 ms
🖥️ 예상 출력 (실패 — 통신 불가)
PING 192.168.1.123 (192.168.1.123) 56(84) bytes of data. From 192.168.1.50 icmp_seq=1 Destination Host Unreachable From 192.168.1.50 icmp_seq=2 Destination Host Unreachable # 또는 모두 시간 초과: 5 packets transmitted, 0 received, 100% packet loss
💡 출력을 읽는 법 — 핵심 지표
  • "64 bytes from X.X.X.X" 줄: 카메라가 응답함 (✅ 정상)
  • "time=X.X ms": 응답 시간. 직결이라 0.5ms 이하가 정상. 1초 이상이면 문제 있음
  • "5 received, 0% packet loss": 100% 응답 = 완벽한 통신 ✅
  • "Destination Host Unreachable": 카메라까지 도달 못 함 (단계 4~6 다시)
  • "100% packet loss": 카메라가 응답 안 함 (카메라 전원 또는 IP 확인)
  • "ttl=64": Time To Live. 패킷이 거쳐갈 수 있는 라우터 수. 직결이라 그대로 64
📞
쉽게 말하면

"전화 한 번 걸어보기"입니다. 카메라에게 5번 전화를 걸고 5번 다 "여보세요" 응답이 오면 통신 성공. 만약 통화가 안 되면 위의 모든 설정이 무의미해요. 그래서 RTSP 영상 단계로 가기 전에 반드시 ping부터 확인합니다.

🎬
RTSP URL 검증
카메라가 어떤 영상 주소를 지원하는지 확인
🤔 왜 URL을 여러 개 시도하나요?
카메라마다 영상을 받을 수 있는 RTSP URL 형식이 다릅니다. 표준이 명확하지 않아서 제조사마다 자기 방식을 씁니다. Hikvision은 /Streaming/Channels/101, anjvision은 /stream1, 일부 구형은 /h264/ch1/main/av_stream 등. 중국 OEM 카메라(예: 빅맥님 카메라)는 두 가지 패턴 모두 지원하기도 합니다. 어떤 URL이 작동하는지 ffprobe로 미리 확인하는 것이 안전합니다.

먼저 ffmpeg(ffprobe 포함) 설치 확인:

⌨️ 설치 확인
$ffprobe -version

설치 안 되어 있다면:

⌨️ ffmpeg 설치 (ffplay와 ffprobe 포함)
$sudo apt install ffmpeg -y

▶ 가장 흔한 URL 먼저 시도 (Hikvision 호환 메인 스트림)

⌨️ ffprobe로 URL 검증 (영상은 안 띄움, 정보만)
$ffprobe -rtsp_transport tcp \ rtsp://admin:123456@192.168.1.123:554/Streaming/Channels/101
📖 명령어 분해 — RTSP URL 구조
ffprobe"파일이나 스트림의 정보를 분석"하는 도구. 영상은 안 띄우고 정보만 출력. 검증용으로 가벼움.
-rtsp_transport tcpRTSP를 TCP로 받으라는 의미. UDP는 패킷 손실로 영상 끊김. TCP가 안정적 (살짝 느림).
rtsp://프로토콜. "RTSP 방식으로 받겠다"는 의미.
admin:123456@사용자명:비밀번호. 카메라 웹 UI 로그인 정보와 동일. @는 "이 계정으로"의 의미.
192.168.1.123카메라 IP (단계 2에서 발견).
:554RTSP 표준 포트. 거의 모든 카메라가 이 포트 사용.
/Streaming/Channels/101경로. 101=메인 스트림(1080p), 102=서브 스트림(480p).
🖥️ 예상 출력 (성공)
Input #0, rtsp, from 'rtsp://admin:***@192.168.1.123:554/Streaming/Channels/101': Metadata: title : RTSP/RTP stream from anjvision ipcamera Duration: N/A, start: 0.000000, bitrate: N/A Stream #0:0: Video: h264 (High), yuv420p, 1920x1080, 25 fps, 25 tbr Stream #0:1: Audio: pcm_mulaw, 8000 Hz, mono, s16, 64 kb/s
🖥️ 예상 출력 (실패 — 401 인증 오류)
[rtsp @ 0x7b360c000c80] method DESCRIBE failed: 401 Unauthorized rtsp://admin:wrong@192.168.1.123:554/Streaming/Channels/101: Server returned 401 # 원인: 비밀번호 오류 또는 사용자명 오류
🖥️ 예상 출력 (실패 — URL 경로 오류)
[rtsp @ 0x7b360c000c80] method DESCRIBE failed: 404 Stream Not Found # 원인: 이 카메라는 다른 URL 패턴 사용 → /stream1 시도
💡 출력을 읽는 법
  • "Input #0, rtsp, from ...": ffprobe가 성공적으로 연결됨
  • "title: RTSP/RTP stream from ...": 카메라 제조사 자기소개 (anjvision/Hikvision 등)
  • "Video: h264, 1920x1080, 25 fps": 영상 사양. 1080p 25프레임 = 좋은 화질
  • "401 Unauthorized": 비밀번호 틀림
  • "404 Stream Not Found": URL 경로 틀림 → 다른 패턴 시도
🔄
실패 시 시도해볼 다른 URL 패턴:
  • /Streaming/Channels/102 (서브 스트림 480p)
  • /stream1 (anjvision 자체 메인)
  • /stream2 (anjvision 자체 서브)
  • /h264/ch1/main/av_stream (구형 Hikvision)
각 URL을 차례로 ffprobe로 검증하여 작동하는 것을 찾으세요.
✅ "1920x1080" 같은 해상도 정보가 보였나요?  → 단계 9에서 실제로 영상 보기
9
📺
ffplay로 실시간 영상 시청
🎉 드디어 영상을 화면에 띄우는 단계
🤔 마지막 단계 — 무엇을 하나요?
단계 8에서 검증한 RTSP URL을 ffplay로 열어 NUC 화면에 영상 창을 띄웁니다. ffprobe는 정보만 알려주지만 ffplay는 실제로 영상을 디코딩해서 보여줍니다. 이 단계가 성공하면 셋업 완료 — 카메라에서 실시간 영상이 NUC으로 흐르고 있는 것이 확인됩니다.
⌨️ 영상 띄우기
$ffplay -rtsp_transport tcp \ -fflags nobuffer -flags low_delay -framedrop -an \ rtsp://admin:123456@192.168.1.123:554/Streaming/Channels/101
📖 명령어 분해 — 옵션별 의미
ffplay"FFmpeg의 영상 플레이어". ffmpeg 패키지에 포함된 가벼운 비디오 플레이어. VLC보다 단순하지만 충분.
-rtsp_transport tcpUDP 대신 TCP로 RTSP 받기. 끊김 없는 영상.
-fflags nobuffer입력 버퍼링 끄기. 실시간성 ↑ (지연 감소).
-flags low_delay디코더의 저지연 모드. CCTV/모니터링에 필수.
-framedrop처리 못 따라잡으면 프레임을 떨어뜨림 (영상이 밀리지 않게).
-an"audio no". 오디오 끄기. 모니터링은 보통 영상만 필요. 시끄러움 방지.
🖥️ 예상 출력 (콘솔)
Input #0, rtsp, from '...': Stream #0:0: Video: h264 (High), yuv420p, 1920x1080, 25 fps # ↑ 그리고 별도 창이 뜨면서 실시간 영상 표시! 2.34 A-V: 0.000 fd= 0 aq= 0KB vq= 256KB sq= 0B
🎬
창 조작법:
• Q 또는 ESC: 종료
• F: 전체 화면 전환
• 마우스로 창 크기 조절 가능
• 콘솔에서 Ctrl + C로도 종료 가능
💡 영상이 잘 나오는지 체크
  • 창이 뜨고 영상이 부드럽게 재생됨 → 셋업 완전 성공 🎉
  • 영상이 깜빡거리거나 끊김 → -rtsp_transport tcp가 빠졌는지 확인
  • 창은 뜨는데 검은 화면 → 카메라 렌즈 캡 또는 어두운 환경. 카메라를 밝은 곳으로
  • "A-V: 0.000" 표시: 영상-오디오 동기화 시간. -an 덕분에 0.0
🎉

1. nmcli device status → 이더넷 인터페이스 이름 확인 (예: eno1)
2. sudo tcpdump -i eno1 ... → 카메라 IP 발견 (예: 192.168.1.123)
3. ip addr show eno1 → 현재 NUC IP 상태 파악
4. sudo nmcli connection modify → 영구 IP 설정 (192.168.1.50/24)
5. nmcli connection down/up → 설정 적용
6. ip addr show eno1 | grep 'inet ' → IP 적용 확인
7. ping -c 5 192.168.1.123 → 통신 확인 (5/5 응답)
8. ffprobe ... → RTSP URL 검증 (해상도 확인)
9. ffplay ... → 영상 시청 성공 🎉

자주 막히는 부분 — 트러블슈팅

단계별로 발생할 수 있는 문제와 해결법

증상가능한 원인해결
nmcli 결과에 eno1이 없음다른 이더넷 이름 (eth0, enp3s0 등)TYPE이 ethernet인 줄에서 DEVICE 이름 확인
tcpdump에 아무것도 안 나옴카메라가 이미 부팅 완료 (브로드캐스트 끝남)카메라 전원 OFF → 잠시 대기 → ON 후 재실행
tcpdump "permission denied"sudo 누락명령어 앞에 sudo 붙이기
nmcli modify "unknown connection"프로필 이름 오타 또는 띄어쓰기단계 1 출력의 CONNECTION 컬럼 정확히 복사, "유선 연결 1" (공백 한 칸)
nmcli up "activation failed"케이블 미연결 또는 다른 충돌RJ45 LED 확인, 케이블 다시 꽂기
단계 6에 IP가 안 보임NetworkManager가 덮어씀단계 4의 manual 설정 재확인, 단계 5 재실행
ping "Destination Host Unreachable"NUC과 카메라가 다른 대역단계 4의 IP가 카메라와 같은 대역인지 확인
ping 100% packet loss카메라 전원/케이블 문제RJ45 LED + 카메라 12V 어댑터 확인
ffprobe 401 Unauthorized비밀번호 오류브라우저로 http://192.168.1.123 로그인 시도해 정확한 비번 찾기
ffprobe 404 Stream Not FoundURL 경로가 이 카메라에 안 맞음/stream1, /stream2, /Streaming/Channels/101 등 차례로 시도
ffplay 창이 떴다가 바로 닫힘RTSP 인증 실패 또는 디코딩 실패콘솔 에러 메시지 확인, ffprobe로 먼저 검증
영상이 끊겨서 나옴UDP 사용 중-rtsp_transport tcp 옵션 확인

🎓 이번 실습에서 배운 핵심 개념

개념실제 사용된 곳
네트워크 인터페이스nmcli device status에서 본 eno1, wlp..., wlx...
tcpdump 패킷 청취카메라 IP 자동 발견 (단계 2)
ARP 프로토콜같은 망 안의 IP-MAC 매핑 (tcpdump 출력)
NetworkManager 영구 설정재부팅에도 유지되는 IP (단계 4)
서브넷 / 같은 대역192.168.1.x — 직접 통신 가능한 범위
ping (ICMP)네트워크 통신 기본 검증 (단계 7)
RTSP 프로토콜IP 카메라 영상 스트리밍 표준
RTSP URL 구조rtsp://계정:비번@IP:포트/경로
TCP vs UDP for RTSPTCP는 안정, UDP는 빠름 (CCTV는 TCP)
ffprobe / ffplay영상 검증과 시청 도구
