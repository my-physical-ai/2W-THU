IP 카메라 라우터 연결

NUC 직결로 먼저 카메라 IP를 라우터 대역으로 변경한 후, 라우터로 옮겨 한 번에 통합 망에 합류시키는 단계별 가이드. 시행착오 없이 첫 시도에 성공하는 흐름입니다.

⏱️ 예상 소요: 60~90분
🎯 난이도: 중급
📚 10단계 (선행 6 + 본작업 4)
💡 V3의 핵심 통찰 — 왜 선행작업이 중요한가?

"카메라를 라우터에 연결한 다음 IP를 찾는다"는 일반적 접근은 카메라가 고정 IP를 유지하고 있으면 라우터 망에서 보이지 않는 함정에 빠집니다.

V3는 더 안전한 순서를 제안합니다: 먼저 NUC 직결로 카메라 웹 UI에 접속하여 IP를 라우터 대역의 고정 IP로 변경 + 비밀번호 강화를 모두 끝낸 후, 카메라를 라우터로 옮깁니다. 이렇게 하면 시행착오 없이 한 번에 성공해요.

🎯 이 실습의 두 단계 (Phase) 구조

🟣 Phase 1 — 선행작업 (NUC 직결 상태)
단계 1~6: 카메라를 NUC에 임시 직결한 상태에서 사전 설정을 완료
  • 카메라 현재 IP 발견
  • NUC을 카메라 대역으로 임시 설정
  • 카메라 웹 UI 접속
  • ⭐ IP를 라우터 대역 고정 IP로 변경
  • 비밀번호 강화 (보안)
🟢 Phase 2 — 본작업 (라우터 통합)
단계 7~10: 카메라를 라우터로 옮기고 한 번에 성공
  • 카메라 케이블 라우터로 이동
  • NUC 라우터 망 연결 확인
  • arp-scan으로 한 번에 발견 ✅
  • RTSP 검증 + 멀티 PC 시청
🎓
학습 효과: 이 흐름은 빅맥님이 실제 경험하신 "카메라 안 보임" 시행착오를 학생들이 겪지 않도록 미리 차단합니다. 분기 처리 없이 깔끔한 선형 진행이 가능해요.

🟣 Phase 1 — 선행작업 (단계 1~6)
NUC 직결 상태에서 카메라 IP와 비밀번호를 미리 설정합니다
1
📋
NUC ↔ 카메라 직결 + 인터페이스 확인
선행작업의 시작 — 카메라를 NUC에 임시 연결
🤔 왜 라우터에 바로 안 꽂고 NUC에 먼저?
카메라가 현재 어떤 IP를 갖고 있는지 모르는 상태에서 라우터에 바로 꽂으면, 카메라가 고정 IP를 유지하는 경우 라우터 망에서 찾을 수 없습니다(빅맥님이 실제 경험하신 함정). 그래서 먼저 NUC과 일대일로 연결한 격리망에서 카메라 IP를 정확히 알고, 우리가 원하는 라우터 대역 IP로 미리 변경한 다음 라우터로 옮기는 것이 훨씬 안전합니다.
⚠️
사전 준비 — 물리적 작업:
① 카메라 12V 어댑터 연결
② 카메라 RJ45 ↔ NUC eno1에 랜선 직접 연결 (라우터 아님!)
③ 카메라 전원 ON → RJ45 LED 녹색 깜빡임 확인
⌨️ 터미널에 입력
$nmcli device status
📖 명령어 분해
nmcliNetworkManager 명령어 도구.
device status모든 네트워크 장치의 현재 상태 출력.
🖥️ 예상 출력
DEVICE TYPE STATE CONNECTION wlp0s20f3 wifi 연결됨 NTH215 wlx002e2d40106e wifi 연결됨 zeta_amr_rnd 3 eno1 ethernet 연결됨 유선 연결 1 lo loopback 연결됨 (외부) lo
💡 확인 사항
  • TYPE이 'ethernet'인 줄의 DEVICE 이름 메모 (보통 eno1)
  • STATE가 '연결됨'이면 OK — 카메라와 케이블 연결 정상
  • WiFi(wlx, wlp)는 무시 — 본작업(Phase 2)에서 사용
📝
메모: 이더넷 인터페이스 이름 ____________ (예: eno1)

2
🔍
카메라 현재 IP 발견 (tcpdump)
카메라가 부팅 시 외치는 신호를 청취
🤔 카메라 IP를 모르는데 어떻게 찾나요?
카메라가 부팅하거나 케이블 연결 직후 자동으로 보내는 ARP/DHCP 브로드캐스트 신호를 tcpdump으로 청취합니다. 카메라는 "저 여기 있어요, IP는 192.168.X.Y에요" 같은 자기소개를 자동으로 합니다. 이를 가로채면 카메라의 현재 IP를 알 수 있어요.
⌨️ 터미널에 입력 (eno1은 단계 1에서 확인한 이름)
$sudo tcpdump -i eno1 -n -e 'arp or (udp port 67 or udp port 68)'
📖 명령어 분해 (자세한 설명은 실습자료 1 V2 단계 2 참조)
sudo tcpdump네트워크 패킷 가로채기 (관리자 권한 필요).
-i eno1어떤 인터페이스에서 들을지 지정.
-n -eIP를 그대로 표시(-n), MAC도 표시(-e).
'arp or ...'ARP 또는 DHCP 패킷만 필터링.
💡
팁: 아무 출력이 없으면 카메라 전원을 잠시 OFF → ON. 부팅 직후 즉시 외치는 신호가 잡힙니다. 30초 정도 후 Ctrl+C로 중단.
🖥️ 예상 출력
listening on eno1, link-type EN10MB (Ethernet) 14:23:01.234 f0:00:00:99:e0:85 > ff:ff:ff:ff:ff:ff, ARP, Request who-has 192.168.1.1 tell 192.168.1.123, length 46 ^C 1 packet captured
💡 출력에서 추출할 정보
  • 카메라 MAC: f0:00:00:99:e0:85 (분홍색)
  • 카메라 현재 IP: tell 뒤의 IP (노란색) = 192.168.1.123
  • 이 IP가 카메라의 현재 위치이며, 단계 3, 4에서 사용
📝
메모:
▸ 카메라 현재 IP: ____________________ (예: 192.168.1.123)
▸ 카메라 MAC: ____________________ (예: f0:00:00:99:e0:85)
✅ 카메라 IP 발견하셨나요?  → 단계 3에서 NUC을 같은 대역으로 설정
3
⚙️
NUC을 카메라 대역으로 임시 IP 설정
웹 UI 접속을 위한 일시적 NUC IP 변경
🤔 NUC IP를 왜 바꿔야 하나요?
카메라가 192.168.1.123에 있고 NUC이 다른 대역(예: 192.168.50.x)이면 직접 통신이 불가능합니다. 카메라 웹 UI에 접속하려면 NUC도 같은 대역(192.168.1.x)에 임시로 들어가야 해요. 이 설정은 "선행작업이 끝나면 곧 원복할 임시 설정"입니다. 본작업(Phase 2)에서 라우터 망으로 갈 때 다시 DHCP로 돌립니다.

▶ 명령어 1: NUC IP를 카메라 대역으로

⌨️ NUC eno1 임시 IP 설정
$sudo nmcli connection modify "유선 연결 1" \ ipv4.method manual \ ipv4.addresses 192.168.1.50/24 \ ipv4.gateway "" \ ipv4.dns ""

▶ 명령어 2: 설정 적용

⌨️ 재연결로 적용
$sudo nmcli connection up "유선 연결 1"

▶ 명령어 3: 통신 확인

⌨️ 카메라 ping
$ping -c 3 192.168.1.123
🖥️ ping 예상 출력 (성공)
PING 192.168.1.123 (192.168.1.123) 56(84) bytes of data. 64 bytes from 192.168.1.123: icmp_seq=1 ttl=64 time=0.5 ms 64 bytes from 192.168.1.123: icmp_seq=2 ttl=64 time=0.4 ms 64 bytes from 192.168.1.123: icmp_seq=3 ttl=64 time=0.5 ms 3 packets transmitted, 3 received, 0% packet loss
💡 결과 해석
  • "3 received, 0% packet loss" → 완벽한 통신 ✅
  • "Destination Host Unreachable" → 카메라 IP 오류 (단계 2 재확인)
  • "100% packet loss" → NUC IP 설정 미적용 (명령어 1, 2 재실행)
📖
자세한 설명 필요시: 이 단계의 명령어들에 대한 자세한 옵션 분해와 의미는 실습자료 1 V2의 단계 4~7에 풀이되어 있습니다. 동일한 명령어이므로 거기를 참조하세요.
✅ ping 성공하셨나요?  → 단계 4에서 카메라 웹 UI 접속
4
🔐
브라우저로 카메라 웹 UI 접속
관리 페이지 진입 — 단계 5, 6의 변경 작업 준비
🤔 명령어가 아니라 웹 UI를 쓰나요?
카메라의 IP 설정과 비밀번호 변경은 명령어가 아닌 카메라 자체의 웹 인터페이스를 통해서만 가능합니다. 카메라는 서버 역할을 하며, 브라우저가 클라이언트가 되어 카메라 내부 설정을 변경하는 방식이에요. 대부분의 IP 카메라가 이 방식을 사용합니다.
⌨️ 브라우저 실행 (Firefox/Chrome 등)
$firefox http://192.168.1.123 & # 또는 브라우저 직접 열고 주소창에 입력: http://192.168.1.123
📖 URL 구조
http://웹 프로토콜 (HTTPS가 아닌 HTTP — 카메라 내부 통신).
192.168.1.123카메라 현재 IP (단계 2에서 발견).
&백그라운드 실행 — 터미널을 다시 사용 가능하게 함.
🖥️ 브라우저 화면 (예상)
┌─────────────────────────────────┐ │ ╔═══════════════════════════╗ │ │ ║ Camera Login ║ │ │ ║ ║ │ │ ║ Username: [admin ] ║ │ │ ║ Password: [..........] ║ │ │ ║ ║ │ │ ║ [ Login ] ║ │ │ ╚═══════════════════════════╝ │ └─────────────────────────────────┘ # 카메라 모델마다 다르지만 비슷한 로그인 화면

✅ 로그인 성공 → 카메라 관리 페이지 진입?  → 단계 5에서 IP 변경 ⭐
5
⭐
⭐ 카메라 IP를 라우터 대역 고정 IP로 변경
V3의 핵심 단계 — 라우터로 이동하기 전에 미리 IP 변경
🤔 이 단계가 V3의 핵심인 이유
카메라가 지금 192.168.1.123이라는 옛 IP를 갖고 있는데, 라우터 망은 192.168.50.x 대역입니다. 카메라가 이 옛 IP를 유지한 채 라우터에 연결되면 라우터와 통신이 안 되어 찾을 수 없습니다 (빅맥님이 V1, V2에서 경험하신 함정).

V3의 해결책: 카메라가 아직 NUC 직결 상태일 때 IP를 미리 라우터 대역의 고정 IP(예: 192.168.50.123)로 바꿔둡니다. 그러면 라우터에 연결되자마자 정상적으로 통신 가능합니다.
🚚
쉽게 말하면

"이사 가기 전에 새 주소를 미리 등록"하는 것입니다. 옛 집(192.168.1.123)에 살면서 새 동네(192.168.50.x) 주소(192.168.50.123)를 미리 정해놓고, 그 다음 새 집으로 이사해요. 그러면 이사하자마자 새 주소로 모든 우편이 정상 도착합니다.

⚠️
중요한 사전 정보 확인: 카메라를 옮길 라우터의 IP 대역을 미리 알아야 합니다. 보통 라우터 관리 페이지에서 확인 가능하며, 흔한 대역은 192.168.0.x, 192.168.1.x, 192.168.50.x 등입니다.

▶ 작업 1: 카메라 웹 UI에서 네트워크 설정 메뉴로 이동

메뉴 경로: 시스템(System) → 네트워크(Network) → IPv4 설정

※ 카메라 모델에 따라 메뉴명이 약간 다를 수 있음: "네트워크 → TCP/IP 설정", "환경설정 → 네트워크" 등

▶ 작업 2: 고정 IP 모드로 설정

📋 설정값 입력 (라우터 대역에 맞춰서)
시스템 → 네트워크 → IPv4 설정 IP 모드: 정적 / Static / 수동 # DHCP가 아님! IP 주소: 192.168.50.123 # ⭐ 라우터 대역 (50)으로 변경 서브넷 마스크: 255.255.255.0 게이트웨이: 192.168.50.1 # 라우터 IP (보통 .1) 주 DNS: 192.168.50.1 # 또는 8.8.8.8 (구글) 보조 DNS: 8.8.4.4 # 선택사항 [저장 / Save]
📖 각 항목의 의미
IP 모드: 정적DHCP가 아닌 우리가 지정한 IP를 영구 사용. 라우터가 다른 IP를 부여하려 해도 거부.
IP 주소: 192.168.50.123라우터 대역(50)에 속한 IP. 끝자리(123)는 다른 기기와 겹치지 않게 선택. 100~200 범위가 안전.
서브넷 마스크: 255.255.255.0"/24"의 다른 표현. 같은 대역의 범위 (192.168.50.0~255).
게이트웨이: 192.168.50.1외부로 나갈 때 거치는 라우터의 IP. 대부분 .1번.
DNS도메인을 IP로 변환하는 서버. 라우터(.1) 또는 구글(8.8.8.8) 사용.

▶ 작업 3: 저장 → 카메라 자동 재부팅

⏳
저장 직후 일어나는 일:
  1. 카메라가 새 IP 적용을 위해 자동 재부팅 (30초~1분)
  2. 재부팅 후 카메라는 192.168.50.123으로 변경됨
  3. 현재 브라우저 페이지는 더 이상 접속 안 됨 (정상)
  4. NUC도 192.168.1.50/24이라 새 IP와 통신 불가 (정상)

▶ 작업 4 (선택): 새 IP로 ping 테스트

새 IP(192.168.50.123)는 NUC의 현재 IP(192.168.1.50)와 다른 대역이라 현재는 ping이 안 됩니다. 이는 정상이에요. 카메라가 변경된 IP를 잘 받았는지 확인하려면 NUC을 잠시 192.168.50.50으로 바꾸고 ping할 수 있지만, 본작업(Phase 2)에서 어차피 확인할 것이라 생략해도 됩니다.

⚠️ 비밀번호 변경 (필수)
선행작업의 마지막 — 라우터로 옮기기 전에 보안 강화
🤔 왜 라우터로 옮기기 전에 비번을 변경?
라우터에 연결되는 순간 카메라는 같은 망의 모든 기기로부터 접근 가능해집니다. 만약 공장 초기 비밀번호(123456) 그대로라면 같은 WiFi에 접속한 누구나 카메라 영상을 볼 수 있어요. NUC 직결 상태(격리망)일 때 미리 비밀번호를 강화해두면, 라우터에 옮긴 직후부터 보안된 상태로 시작합니다.
⚠️
비번 변경 전 NUC IP 임시 원복: 단계 5에서 카메라가 192.168.50.123으로 변경되었으므로, 웹 UI 재접속을 위해 NUC을 같은 대역으로 잠시 바꿔야 합니다. 또는 NUC을 192.168.50.x로 임시 설정 후 새 IP로 접속.

대안: 단계 5 직전에(IP 변경 화면 진입 직후) 같은 페이지에서 비밀번호도 함께 변경 → 한 번에 끝내기.

▶ 권장 방법: 단계 5와 6을 한 번에 처리

💡
효율적인 작업 순서:
  1. 현재 IP(192.168.1.123)로 웹 UI 접속 중일 때
  2. 먼저 비밀번호 변경 (시스템 → 사용자 관리)
  3. 그 다음 IP 변경 (시스템 → 네트워크)
  4. 저장 → 카메라 재부팅 → IP 변경 + 비번 변경 동시 적용

▶ 비밀번호 변경 작업

메뉴 경로: 시스템(System) → 사용자 관리(User Management) → admin → 비밀번호 변경

📋 강력한 비밀번호 권장 요건
✅ 길이 : 12자 이상 ✅ 문자 종류 : 대문자 + 소문자 + 숫자 + 특수문자 모두 포함 ✅ 회피 : 사전 단어, 생일, 전화번호, "password", "admin", "123456" ✅ 예시 형식 : ZetaR0bot!2026 # 실제로는 본인만 아는 다른 비번 사용 ❌ 나쁜 예 : 12345678, password123, admin2026, qwerty
💡 비밀번호 강도 측정
  • 12자 + 4가지 문자 종류 = 해킹 난이도 매우 높음
  • brute force 시간: 8자 = 몇 시간, 12자 = 수만 년
  • 사전 공격: 일반 단어는 즉시 깨짐 → 무작위 조합 필수
  • 비밀번호 관리: Bitwarden, 1Password 같은 도구 사용 권장

카메라를 라우터 LAN으로 이동
물리적 케이블 이동 + NUC 원복
🤔 이제 라우터로 옮겨도 안전한 이유
Phase 1에서 카메라의 IP를 라우터 대역(192.168.50.123)으로 미리 변경했기 때문에, 이제 라우터에 연결되는 즉시 정상적으로 통신 가능합니다. 카메라가 "다른 동네 거주자"가 아니라 이제 "같은 동네 입주자"가 된 상태예요.

▶ 물리 작업 1: 카메라 케이블 이동

🔌
케이블 이동:
  1. 카메라 RJ45에서 NUC eno1으로 가던 랜선을 NUC에서 분리
  2. 같은 랜선을 라우터의 LAN 포트에 꽂기
  3. WAN 포트(보통 색이 다름)와 혼동 주의!
⚠️
WAN vs LAN 구분: 일반적인 라우터:
• WAN 포트: 1개, 보통 파란색 또는 노란색, 인터넷 회선이 들어옴
• LAN 포트: 4개, 모두 같은 색, 가정 내부 기기 연결용
→ 카메라는 반드시 LAN 포트에 꽂기

▶ 명령어 작업: NUC IP를 DHCP 자동으로 원복

Phase 1에서 NUC eno1을 192.168.1.50으로 임시 설정했어요. 이제 본래 DHCP 자동 모드로 원복합니다.

⌨️ NUC eno1 DHCP로 원복
$sudo nmcli connection modify "유선 연결 1" \ ipv4.method auto \ ipv4.addresses "" \ ipv4.gateway "" \ ipv4.dns ""
📖 변경 사항
ipv4.method auto"manual" → "auto" 변경. DHCP로 자동 IP 받기.
ipv4.addresses ""수동 IP 삭제 (빈 값으로).
ipv4.gateway / dns ""마찬가지로 빈 값으로 원복.
⌨️ eno1 비활성화 (사용 안 함)
$sudo nmcli connection down "유선 연결 1"
💡 NUC eno1 원복이 필요한 이유
  • eno1을 192.168.1.50으로 유지하면 다음 사용 시 충돌 가능
  • DHCP 자동으로 두면 어떤 환경에 옮겨도 호환됨
  • eno1 자체는 케이블이 없으니 "사용할 수 없음" 상태가 됨 (정상)
✅ 카메라 라우터 이동 + NUC eno1 원복?  → 단계 8

📡
NUC의 라우터 망 연결 확인
NUC이 어떤 인터페이스로 라우터에 연결됐는지 확인
🤔 NUC의 라우터 망 상태를 왜 확인?
카메라가 라우터 망(192.168.50.x)에 있으니, NUC도 같은 라우터 망에 있어야 통신 가능합니다. NUC이 WiFi로 라우터에 연결되어 있다면 자동으로 같은 망에 있을 거예요. 이 단계에서는 그 상태를 확인하고, 단계 9의 스캔에 사용할 인터페이스 이름을 파악합니다.

▶ 명령어: NUC 인터페이스 상태 확인

⌨️ 터미널에 입력
$nmcli device status
🖥️ 예상 출력 (빅맥님 환경)
DEVICE TYPE STATE CONNECTION wlp0s20f3 wifi 연결됨 NTH215 wlx002e2d40106e wifi 연결됨 zeta_amr_rnd 3 lo loopback 연결됨 (외부) lo eno1 ethernet 사용할 수 없음 --
💡 확인 사항
  • 라우터 SSID (예: zeta_amr_rnd 3)에 연결된 인터페이스 찾기
  • 위 예시에서는 wlx002e2d40106e가 라우터에 WiFi로 연결됨
  • eno1은 케이블 없어 "사용할 수 없음" (Phase 1에서 카메라 옮긴 후 정상 상태)
  • 이 인터페이스 이름을 단계 9에서 사용

▶ NUC IP 대역 확인

⌨️ 라우터 망 IP 대역 확인
$ip addr show wlx002e2d40106e | grep 'inet '
🖥️ 예상 출력
inet 192.168.50.237/24 brd 192.168.50.255 scope global dynamic noprefixroute wlx002e2d40106e
💡 결과 해석
  • NUC IP: 192.168.50.237 → 라우터 대역(50)에 정상 진입
  • 이제 카메라(192.168.50.123)와 NUC이 같은 대역에 있음
  • "dynamic" = DHCP로 받은 IP (자동 갱신될 수 있음)
  • 두 기기가 직접 통신 가능한 상태 → 단계 9에서 검증

▸ NUC 라우터 망 IP: ____________________ (예: 192.168.50.237)
▸ 사용할 인터페이스: ____________________ (예: wlx002e2d40106e)

arp-scan으로 카메라 한 번에 발견
Phase 1 효과 검증 — 시행착오 없는 첫 성공
🤔 V3의 진가가 드러나는 단계
V1, V2에서는 이 단계에서 카메라가 안 보이는 함정에 빠졌습니다. 그러나 V3는 Phase 1에서 카메라 IP를 미리 192.168.50.123으로 변경했기 때문에, 이번 스캔에서 한 번에 발견됩니다. 시행착오 없이 깔끔하게 성공해요.

▶ arp-scan 도구 설치 확인 (안 깔려 있으면 설치)

⌨️ arp-scan 설치
$sudo apt install arp-scan -y

▶ 라우터 망 스캔

⌨️ 같은 대역 모든 기기 스캔
$sudo arp-scan --interface=wlx002e2d40106e --localnet
📖 명령어 분해
sudo관리자 권한 (네트워크 카드 직접 접근).
arp-scan같은 대역 전체에 ARP 요청을 보내 응답한 모든 기기 목록 수집.
--interface=...스캔에 사용할 NUC 인터페이스 (단계 8에서 확인).
--localnet"내가 속한 대역" 전체 스캔. NUC IP에서 자동 추출 (예: 192.168.50.0/24).
🖥️ 예상 출력 (V3에서는 한 번에 성공!)
Interface: wlx002e2d40106e, type: EN10MB, MAC: 00:2e:2d:40:10:6e, IPv4: 192.168.50.237 Starting arp-scan 1.10.0 with 256 hosts 192.168.50.1 04:42:1a:68:ec:60 (라우터) 192.168.50.123 f0:00:00:99:e0:85 (카메라! 우리가 설정한 IP) ⭐ 192.168.50.237 00:2e:2d:40:10:6e (NUC 자기 자신) 3 packets received by filter, 0 packets dropped by kernel Ending arp-scan 1.10.0: 256 hosts scanned. 3 responded
💡 V3의 성공 확인
  • 192.168.50.123이 보이면 → Phase 1 성공! ✅
  • 이 IP는 우리가 Phase 1에서 직접 지정한 고정 IP
  • 카메라 MAC(f0:00:00:99:e0:85)이 일치하는지 확인
  • 분기 처리 없이 깔끔하게 한 번에 발견

▶ 추가 검증: ping으로 통신 확인

⌨️ 카메라 ping 검증
$ping -c 3 192.168.50.123
🖥️ 예상 출력
PING 192.168.50.123 (192.168.50.123) 56(84) bytes of data. 64 bytes from 192.168.50.123: icmp_seq=1 ttl=64 time=2.5 ms 64 bytes from 192.168.50.123: icmp_seq=2 ttl=64 time=2.3 ms 64 bytes from 192.168.50.123: icmp_seq=3 ttl=64 time=2.4 ms 3 packets transmitted, 3 received, 0% packet loss
⚡
WiFi vs 직결 응답 시간 차이: 직결 환경(Phase 1)에서는 ping이 ~0.5ms였는데, WiFi 라우터 환경에서는 ~2.5ms로 약간 느림. 정상적인 차이입니다. 영상 시청에는 문제 없음.

👥
RTSP 검증 + ffplay 시청 (멀티 PC)
🎉 라우터 연결의 진정한 장점 발현
🤔 라우터 연결의 진짜 장점
지금부터 만들 RTSP URL은 같은 라우터에 연결된 어떤 PC에서든 동일하게 작동합니다. NUC, 학생 노트북, Mac, Windows PC 모두 같은 URL로 카메라 영상에 직접 접근 가능. 이게 NUC 직결(실습자료 1)과의 결정적 차이입니다.

▶ NUC에서 영상 시청 (먼저 검증)

⌨️ NUC에서 영상 띄우기 (새 IP + 새 비번 사용)
$ffplay -rtsp_transport tcp \ -fflags nobuffer -flags low_delay -framedrop -an \ rtsp://admin:새비번@192.168.50.123:554/Streaming/Channels/101
📖 RTSP URL 분해 (Phase 1에서 변경한 값 반영)
rtsp://RTSP 프로토콜.
admin:새비번사용자명과 단계 6에서 변경한 새 비번.
192.168.50.123단계 5에서 변경한 새 IP (라우터 대역).
:554RTSP 표준 포트.
/Streaming/Channels/101메인 스트림 (1080p). 102는 서브 (480p).

▶ ⭐ 다른 PC에서도 같은 명령어 (라우터 연결의 진가)

같은 WiFi(zeta_amr_rnd)에 연결된 어떤 PC에서든:

⌨️ 학생 노트북 / 다른 PC에서 (Linux/Mac/Windows)
$ffplay -rtsp_transport tcp rtsp://admin:새비번@192.168.50.123:554/Streaming/Channels/101

V3의 최종 가치: 5명의 학생이 각자 노트북에서 동시에 위 명령어를 실행하면 모두 영상이 보입니다. NUC을 거치지 않고 카메라에서 직접 영상을 받아요. 4월 교육에서 OpenCV 영상 처리 실습이 가능한 이유입니다.

"방송국 채널 개설 + 보안 강화 완료"입니다. 카메라가 라우터 대역의 안정적인 고정 IP로 자리 잡았고, 강력한 비밀번호로 보호되며, 라우터 망의 모든 기기가 안전하게 시청할 수 있어요. 시행착오 없이 깔끔하게 완성된 환경입니다.

실습 체크리스트 — V3 흐름

두 Phase로 나누어 단계별 진행 체크

🟣 Phase 1 — 선행작업 (NUC 직결 상태)
1. 카메라 ↔ NUC eno1 직결 + nmcli device status 인터페이스 확인
2. sudo tcpdump으로 카메라 현재 IP 발견 (예: 192.168.1.123)
3. NUC을 같은 대역으로 임시 설정 (192.168.1.50/24) + ping 확인
4. 브라우저로 카메라 웹 UI 접속 (http://192.168.1.123)
5. ⭐ 카메라 IP를 라우터 대역 고정 IP로 변경 (192.168.50.123)
6. 비밀번호 강력하게 변경 (12자 이상 복합)
🟢 Phase 2 — 본작업 (라우터 통합)
7. 카메라 케이블을 NUC → 라우터 LAN으로 이동 + NUC eno1 DHCP 원복
8. nmcli device status + ip addr show로 NUC 라우터 망 IP 확인
9. sudo arp-scan --localnet으로 카메라 발견 (한 번에 성공) + ping
10. ffplay로 영상 시청 + 다른 PC에서도 시청 확인 🎉
